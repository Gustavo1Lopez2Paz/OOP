Programación Orientada a Objetos


¿Que es la POO?

Es un paradigma de programación que utiliza "objetos" para representar datos y usar métodos que operen sobre esos datos.


¿Que es la herencia?

La herencia es una forma de utilizar clases existentes y utilizarlas para crear nuevas clases. Permite heredar tanto atributos como métodos de la clase, ya en la nueva permite añadir mas métodos o atributos e incluso modificar los ya existentes.

  1. Reutilización de Código: Permite reutilizar el código de la clase base en la subclase, lo que evita la duplicación de código y facilita la extensión de la funcionalidad.
  2. Extensibilidad: Las subclases pueden ampliar la funcionalidad de las clases base añadiendo nuevos métodos o atributos específicos. También pueden sobreescribir métodos heredados para cambiar su comportamiento.
  3. Jerarquía de Clases: La herencia establece una jerarquía de clases que refleja relaciones "es un" entre las clases. Por ejemplo, una Perro puede ser una subclase de Animal, indicando que un perro es un tipo de animal.

¿Que es el polimorfismo?

Permite que una unica interfaz se pueda comportar de diferentes maneras segun en el objeto que sea invocado. lo que significa que un metodo al ser utilizado en diferentes objetos puede comportarse de diferentes maneras según su pripia implementación.


  1. Sobrecarga de Métodos: Permite definir múltiples métodos con el mismo nombre pero con diferentes parámetros en una misma clase. Por ejemplo, un método dibujar() puede tener versiones que acepten diferentes tipos de argumentos.
  2. Sobrescritura de Métodos: En las subclases, se pueden redefinir métodos heredados para que se comporten de manera diferente. Esto se hace a través de la sobrescritura, donde el método en la subclase tiene una implementación específica.
  3. Interfaz Común: Permite usar una interfaz común para diferentes tipos de objetos. Por ejemplo, se puede definir una interfaz Forma con un método dibujar(), y diferentes clases como Círculo y Cuadrado pueden implementar esta interfaz con sus propias versiones del método dibujar().

¿Que es la abstracción?

Consiste en simplificar el diseño del software modelando solo las características que deseen ser relevantes del objeto y ocultar los detalles complejos. Con uso de clases abstractas en interfaces, se puede lograr definir una estructura de clases sin especificar detalles de implementación.

  1. Clases Abstractas: Son clases que no pueden ser instanciadas directamente y que pueden contener métodos abstractos (sin implementación). Las clases derivadas deben implementar estos métodos. Las clases abstractas proporcionan una estructura base que las subclases deben completar.
  2. Interfaces: Definen un contrato que las clases deben seguir, especificando métodos que deben ser implementados. Las interfaces no contienen implementación, solo la firma de los métodos.
  3. Modelado de Conceptos: La abstracción permite modelar conceptos del mundo real en términos de clases y objetos, centrándose en las propiedades y comportamientos relevantes para el problema que se está resolviendo.

¿Que es el encapsulamiento?

Sigue un principio de ocultar detalles internos del objeto y solo revelar lo necesario. Para lograr este proceso se utlizan modificadores de acceso que permiten que sólo ciertas partes del código puedan interactuar con datos internos del objeto, protegiendo el estado inteno del objeto a modificaciones no autorizadas.

  1. Control de Acceso: Se utilizan modificadores de acceso (como private, protected y public en muchos lenguajes de programación) para restringir el acceso a los datos y métodos internos. Por ejemplo, los atributos de una clase pueden ser private para que solo los métodos dentro de la clase puedan acceder a ellos.
  2. Interfaces Públicas: La clase proporciona métodos públicos (también conocidos como getters y setters) para interactuar con los datos internos. Estos métodos permiten controlar cómo se accede y modifica la información, manteniendo la integridad del objeto.
  3. Modularidad: El encapsulamiento promueve la modularidad al separar la implementación interna de la interfaz pública. Esto facilita el mantenimiento y la actualización del código sin afectar a otras partes del sistema.
